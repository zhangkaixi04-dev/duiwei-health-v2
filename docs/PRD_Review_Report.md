# 藏真APP - PRD 评审报告

## 一、评审概述
- **评审对象**：《藏真APP - 产品需求文档（PRD）V1.0》
- **评审视角**：高级产品经理 & 技术负责人
- **总体评价**：文档在产品定位、视觉规范和核心交互逻辑上描述得非常详尽，特别是“玻璃拟态”和“花房”的概念极具差异化。然而，在功能闭环、异常流程处理以及技术落地的细节上仍有缺失，建议补充完善以降低开发风险。

## 二、核心问题质询 (Questions)

### 1. 账号与数据体系 (Critical)
- **问题**：文档中未提及“注册/登录”流程。
- **质询**：
    - MVP 版本是否支持多设备同步？如果仅支持本地存储，用户卸载应用或换机后数据如何找回？
    - 如果计划在 V2.0 上线共建功能，V1.0 是否需要预埋账号体系（哪怕是弱账号体系）？
    - **建议**：明确定义 V1.0 的账号策略（如：先游客模式，后绑定手机号）。

### 2. 视觉效果与性能平衡 (Major)
- **问题**：PRD 强调“磨砂玻璃 + 实时模糊 + 10px高斯模糊 + 动效”。
- **质询**：
    - 在中低端 Android 机型上，实时高斯模糊（Backdrop Filter）可能导致严重的掉帧或发热。是否有降级方案（如使用半透明纯色代替模糊）？
    - 动效（花朵绽放）是使用 Lottie、SVGA 还是视频？由于涉及多种花卉和状态，资源包体积如何控制？

### 3. 内容管理闭环 (Major)
- **问题**：文档详细描述了“记录”和“分类修正”，但未提及“编辑”和“删除”。
- **质询**：
    - 用户记录错误（如错别字、选错图）是否支持再次编辑内容？
    - 用户是否可以删除某条藏品？删除后统计数据如何变化？

### 4. 存储与成本 (Minor)
- **问题**：支持图片和音频录制。
- **质询**：
    - 图片和音频是存本地还是云端？如果是云端，对象存储（OSS/S3）的成本预估是多少？
    - 单条记录的图片/音频大小限制是多少？（如：图片是否压缩？音频最长多少秒？）

## 三、文档完善建议 (Suggestions)

### 1. 补充功能模块
建议在“核心功能需求”中补充以下模块的简要逻辑：
- **启动页/引导页**：首次打开 APP 时的品牌传达与权限获取（相机/麦克风/相册）。
- **设置页**：清理缓存、用户协议/隐私政策（合规必备）、版本号。
- **内容编辑/删除**：在“藏品详情页”增加编辑和删除入口。

### 2. 细化交互逻辑
- **异常状态定义**：
    - **无网络**：提交记录时无网络，是本地暂存还是提示失败？
    - **权限拒绝**：点击相机/麦克风时，若用户拒绝权限，如何引导开启？
    - **空状态**：博物馆在没有任何藏品时的展示样式（Empty State）。

### 3. 技术指标明确化
- **图片压缩**：建议明确“上传前压缩至 1080P 或 500KB 以内”。
- **音频限制**：建议限制录音时长（如 60s），避免文件过大。
- **兼容性要求**：明确支持的最低系统版本（iOS/Android），这直接影响玻璃拟态 API 的可用性。

### 4. 逻辑一致性修正
- **V2.0 预埋**：文档提到 V2.0 上线共建博物馆，建议在 V1.0 的数据库设计中预留 `owner_id` (拥有者) 和 `visibility` (可见性) 字段，避免后续重构数据库。
